## 1. Product Vision

PocketClaw is a personal AI assistant for one user (Bryan Tan), running on his
Windows host as a Scheduled Task. It is not a product; it is infrastructure.
The goal is to give one human a single agent that remembers everything he tells
it, ingests the corners of his digital life he can't be bothered to file, and
answers from that knowledge over the messaging apps he already uses.

Architecture follows the Vivian Balakrishnan capture-layer + curation-layer
pattern: capture is vendor-neutral and lossy-on-the-side-of-keep-everything;
curation is on-demand and built on top. The capture layer is the
`KnowledgeBase` (Postgres + pgvector behind `src/modules/knowledge-base/`).
The curation layer is two halves: an Obsidian vault generated by
`src/modules/wiki-generator.ts`, and the agent itself answering recall
queries inline over Telegram, WhatsApp, or the CLI.

The pattern is the durable idea. Implementation details — pgvector vs mnemon,
Claude Code subscription vs Bedrock, TypeScript vs Python ingesters — are
swappable. The PRD is written so it survives the next swap.

Forked from NanoClaw v2. Inherits the host/container/two-DB split, the
OneCLI credential gateway, and the agent-runner sidecar pattern. Diverges
in workspace topology (one always-on `pocketclaw` agent group, not many),
in transport (Claude Code subscription, no Bedrock), and in surface area
(KB + ingestion + photo pipeline, not in trunk NanoClaw).

## 2. User Stories

There is one user. The stories are first-person.

### 2.1 Memory capture

> "I'm walking and just thought of something. I send a Telegram message to
> the bot and forget about it. Three weeks later I ask 'what was that thing
> I said about X' and it tells me, with the date and approximate context."

Wired today. `/memory <text>` is a slash-command in
`groups/pocketclaw/skills/memory/SKILL.md` (rewritten in M5 to call
`kb_remember`; pending the kb-mcp-tool transport from
`.omo/plans/pocketclaw-kb-mcp-tool.md`). Without the MCP tool, plain
inbound messages are still archived by `src/modules/chat-archive.ts` —
recall works; structured tagging via `/memory` is what's pending.

### 2.2 Photo capture

> "I take a photo of a whiteboard, send it to the bot. Later I ask
> 'what was on the whiteboard from the meeting last Tuesday' and it
> describes it back."

Wired today. `src/modules/photo-processor.ts` validates, resizes,
runs the Ollama llava vision model, writes the description into the
KB tagged `source='photo'`. The bytes go to the cleanup queue once
the description is committed. Recall is by semantic search over the
description, not the pixels.

### 2.3 Cloud-source ingestion

> "I don't want to file emails. I want my email to be searchable from
> Telegram by what it was about, not by sender or date."

Partially wired. Cron at 02:00 local runs `src/modules/ingestion/scheduler.ts`
which fans out to Google (`gmail`, `gcal`), Microsoft, Apple, and a few
file watchers. Each ingester is a separate adapter with its own auth path
through OneCLI. SHA256 idempotency on file ingest; `(source, source_id)`
upsert on cloud rows. Output goes to the KB tagged with the source name.
Recall over ingested email is the same shape as recall over chat.

### 2.4 Recall

> "What did Alice say about the budget last month?"

Wired today via the agent's underlying chat archive + ad-hoc semantic
recall. The cleanest form (`/recall <query>`) is in
`groups/pocketclaw/skills/recall/SKILL.md` and depends on the kb-mcp-tool
transport for structured invocation; ad-hoc recall already works because
the agent can describe what it knows from any past conversation in this
session's archive.

### 2.5 Morning digest

> "Every morning at 7am I want a one-message summary: yesterday's email
> highlights, today's calendar, anything I promised someone I'd do."

Aspirational. The 07:00 cron ticks but the handler is parked
(`audit log: morning-digest | SKIP | no-handler`). Re-wire spec lives at
`.omo/plans/pocketclaw-morning-digest-rewire.md`.

### 2.6 Wiki

> "I want a browsable Obsidian vault that auto-organises everything I've
> ever told the bot, by entity (people, projects, topics)."

Aspirational. `src/modules/wiki-generator.ts` works as a pure transform
over the KB; the 03:00 cron that drives it is parked
(`audit log: wiki-regen | SKIP | no-provider`). Re-wire spec at
`.omo/plans/pocketclaw-wiki-cron-rewire.md`. Manual entries via
`/wiki <topic>` (in the skill) work via inline recall + plaintext today.

### 2.7 Document artefacts

> "After a meeting, I send the bot the rough notes and ask for a .docx
> minutes file in my vault."

Aspirational. The three artefact skills (`/minutes` → docx,
`/research` → pdf, `/slides` → pptx) are deferred-with-forward-link.
The host-side generators (`src/modules/meeting-minutes.ts`,
`research-report.ts`, `slide-generator.ts`) exist; the container-side
MCP tool family that drives them does not. Spec at
`.omo/plans/pocketclaw-agent-side-docx-pipeline.md`.

## 3. Goals and Non-Goals

### 3.1 Goals

1. **One agent, always on.** A single `pocketclaw` agent group, one
   container, reachable from any wired channel. No per-task agents.
2. **Capture that doesn't ask permission.** Anything the user sends to
   the bot is stored without confirmation prompts. Stickers and
   reactions are silently dropped (debouncer policy). Everything else
   ends up in the KB.
3. **Recall over the channels he already uses.** Telegram, WhatsApp,
   and the CLI today. The reply is just a chat message; no separate UI.
4. **Curation on top, not in front.** The Obsidian vault and the
   morning digest are derived views, regenerable from the KB. They
   are nice-to-have, not load-bearing. Capture is load-bearing.
5. **Survive a vendor swap.** The PRD, the `KnowledgeBase` interface,
   and the OneCLI auth boundary are written so a future migration off
   pgvector or off Claude Code is mechanical, not a rewrite.
6. **Run unattended on a Windows host.** Scheduled Task at boot.
   Recovery without the user re-authing every Monday.

### 3.2 Non-Goals

1. **Multi-tenant.** No second user. No "team plan". The auth model is
   simple because there is one identity.
2. **Public API.** No HTTP endpoint exposed beyond localhost. Channels
   are the API.
3. **In-chat configuration UI.** Wiring is `ncl` from the host shell.
   The agent does not edit its own group config (yet — see
   self-modification in the inherited NanoClaw doc, currently scoped
   to package install + MCP server registration only).
4. **Native mobile app.** The mobile surface is "Telegram on his phone".
5. **Web dashboard.** The Obsidian vault is the dashboard.
6. **Long-form generation as a primary path.** The agent is a memory
   front-end first and a writing tool second. Document artefacts are
   a side-effect of the recall path, not the headline feature.

## 4. Success Metrics

Each metric is tagged `[instrumented]` if a counter, log line, or test
exists today, `[aspirational]` otherwise. Single-user product → numbers
are about reliability, not scale.

### 4.1 Capture reliability

- **Inbound message → KB row** within 5s of the debouncer flush.
  `[instrumented]` via `src/modules/chat-archive.ts` audit log.
- **Photo → KB description** within 30s of upload (Ollama llava p95
  ~12s on the host's GPU). `[aspirational]` — no timing histogram yet,
  only success/fail audit.
- **Cloud-source idempotency**: re-running the 02:00 ingestion cron
  twice in a row produces zero duplicate rows (verified via
  `(source, source_id)` upsert key + SHA256 file dedup).
  `[instrumented]` via the upsert constraint itself; no test asserts it.

### 4.2 Recall quality

- **`kb.recall(query, k=5)` returns at least one relevant row** for
  any query whose subject was mentioned in chat in the last 90 days.
  `[aspirational]` — no eval harness; gut-check only.
- **Embedding model stability**: `embed_model` column is recorded per
  row so a re-embed pass can target stale rows. `[instrumented]` via
  `src/db/postgres-migrations/001_init.sql`.

### 4.3 Uptime

- **Host uptime ≥ 99% over a rolling week**, measured by Scheduled
  Task last-run timestamps. `[aspirational]` — no monitor, but the
  Scheduled Task auto-restarts on crash.
- **Container respawn within 30s** of a kill, via the `onExit` callback
  pattern in `src/container-runner.ts`. `[instrumented]` via existing
  NanoClaw kill+respawn integration tests (inherited).

### 4.4 Cost

- **Claude Code subscription** is the LLM line item; Bedrock is gone.
  No per-token billing surprises. `[instrumented]` via the absence of
  `CLAUDE_CODE_USE_BEDROCK` and AWS env vars in `.env.sample`.
- **Embedding cost = $0.** Ollama runs locally (host GPU).
  `[instrumented]` via the Ollama provider config.

### 4.5 Trust

- **Zero credential leaks into chat context.** OneCLI gateway is the
  sole credential surface; the agent never sees raw tokens.
  `[instrumented]` via the `onecli-gateway` skill convention + audit.
- **Photo bytes deleted after description.** `[instrumented]` via the
  cleanup step in `src/modules/photo-processor.ts`.

## 5. Competitive Analysis

PocketClaw is single-user personal infra; "competitive" here means
"prior art the design borrows from or deliberately diverges from".
Five reference points:

### 5.1 NanoClaw v2 (parent)

PocketClaw is a fork. Inherits the host/container/two-DB session
split, the OneCLI credential gateway, the agent-runner sidecar, the
`ncl` admin CLI, and the channel adapter pattern. See `CLAUDE.md`
for the inherited architecture.

Diverges in:

- **Workspace topology.** NanoClaw is many-agent-groups, many-DMs,
  many-users. PocketClaw is one always-on `pocketclaw` agent group,
  one user (`telegram:bryanb_t`, `whatsapp:6592348112`, etc), one
  Obsidian vault.
- **Provider.** NanoClaw default ships Claude Code subscription;
  PocketClaw locks that in (no Bedrock fallback, no AWS env vars).
- **KnowledgeBase.** NanoClaw has no persistent cross-session memory
  beyond per-session DBs. PocketClaw adds the `KnowledgeBase` module
  family and its Postgres+pgvector backing store as first-class.
- **Ingestion.** NanoClaw has no scheduled ingestion. PocketClaw
  adds `src/modules/ingestion/` with five adapter families and a
  fault-isolated scheduler.
- **Photo pipeline.** Same.

### 5.2 ChatGPT memory

ChatGPT's "memory" feature is the closest mass-market analogue.
Rejected as the substrate because:

- **Cloud-only.** PocketClaw stores in local Postgres; the user owns
  the bytes.
- **No source attribution.** ChatGPT memory does not preserve
  "where did I learn this". The `KnowledgeBase` row schema preserves
  `source` and `source_id` so a recall can cite Gmail / a specific
  message-id / a photo file.
- **No structured ingestion.** ChatGPT cannot watch a Gmail account
  on a 24h cadence and fold the bodies into the same memory.

### 5.3 Obsidian (curation reference)

PocketClaw uses Obsidian-format Markdown as its curation surface,
not as the source of truth. `src/modules/wiki-generator.ts` writes
WikiLinks; the user reads with Obsidian Desktop.

Why not store IN Obsidian: SHA-keyed semantic search over a vault
of unstructured notes is slower and lossier than a real vector DB
with row-level metadata. The KB is the truth; the vault is a view.

### 5.4 Hippocampus / Letta / mem0 (memory-frameworks)

Considered for the capture layer. Rejected for adding a layer of
opinions (entity extraction strategy, reflection cadence, prompt
shape) that should belong to the curation half. The
`KnowledgeBase` interface in `src/modules/knowledge-base/index.ts`
is intentionally bare: insert with optional tags, recall by query
or entity. Higher-level memory operations (prioritisation, decay,
synthesis) are jobs for the curation layer or for future skills,
not for the substrate.

### 5.5 Bedrock + mnemon (predecessor stack)

The previous PocketClaw stack used AWS Bedrock for inference and a
custom Python service ("mnemon") for memory. Both removed during
the knowledge re-arch (`.omo/plans/pocketclaw-knowledge-rearch.md`,
phases P1–P7). Reasons:

- Bedrock added per-token billing and AWS auth ceremony for a
  single-user product. Claude Code subscription is a flat fee.
- mnemon was a separate Python process with its own auth, its own
  schema migrations, and its own failure modes. Replacing it with
  pgvector behind the `KnowledgeBase` interface eliminated a whole
  service and its IPC.

The mnemon experience taught the durable lesson: the substrate
should be boring. Postgres + pgvector + a 50-line Ollama embed
client is boring enough.

## 6. System Architecture

Two diagrams — one for the inherited host/container/two-DB pattern,
one for the PocketClaw-specific capture+curation overlay.

### 6.1 Host / container / two-DB (inherited from NanoClaw v2)

```
                      ┌─────────────────────────────────┐
                      │  Host (Node, single process)    │
                      │                                 │
  Telegram ──┐        │  channels/  ──▶ router.ts       │
  WhatsApp ──┼──msg──▶│                  │              │
  CLI       ─┘        │                  ▼              │
                      │  session-manager.ts             │
                      │       │                         │
                      │       ▼                         │
                      │  data/v2-sessions/<id>/         │
                      │   ├── inbound.db   (host write) │
                      │   └── outbound.db  (host read)  │
                      │       ▲           │             │
                      │       │           │             │
                      │  delivery.ts ◀────┘             │
                      └───────│─────────────────────────┘
                              │ (DB files mounted)
                      ┌───────▼─────────────────────────┐
                      │  Container (per agent group)    │
                      │                                 │
                      │   agent-runner/  ──▶ Claude     │
                      │     poll inbound.db             │
                      │     write outbound.db           │
                      │     MCP tools (kb_*, onecli)    │
                      └─────────────────────────────────┘
```

Key invariants (inherited; restated for completeness):

- **Two DB files per session.** Host writes only `inbound.db`,
  container writes only `outbound.db`. Exactly one writer per file
  → no cross-mount lock contention.
- **No IPC besides DB rows.** No stdin piping, no HTTP socket, no
  file watcher. A heartbeat is a `touch /workspace/.heartbeat`.
- **Even/odd `seq`.** Host even, container odd. Trivial conflict
  detection.
- **Container = Docker** on this Windows host (Apple containers
  selected on macOS via `src/container-runtime.ts`).

### 6.3 Component boundaries

Five layers, each with one job:

1. **Channel adapters** (`src/channels/`). Translate platform-specific
   protocol to the internal `Message` shape. Plug into the registry;
   skill-installed from the `channels` branch.
2. **Routing + entity model** (`src/router.ts`, `src/db/`).
   Resolve `(platform, chat_id, sender)` → user → messaging-group →
   agent-group → session.
3. **Capture** (`src/modules/{chat-archive, photo-processor,
   ingestion/*}.ts`). Anything that lands new bytes in the KB.
4. **Substrate** (`src/modules/knowledge-base/*`, Postgres). Store,
   embed, retrieve. Vendor-neutral via the `KnowledgeBase` interface.
5. **Curation** (`src/modules/wiki-generator.ts`, agent recall via
   `kb_*` MCP tools, planned cron handlers for digest and wiki).
   Read-only over the substrate; outputs are regenerable views.

### 6.4 The OneCLI credential boundary

OneCLI is the only thing on the host that holds raw credentials.
Every credentialed action a tool wants to take goes through the
OneCLI proxy, with secrets injected at request time. The agent
never sees the credential value. Agent-side: container skill
`container/skills/onecli-gateway/SKILL.md` teaches the agent how
the proxy works and what auth-error shapes mean. Host-side:
`src/onecli-approvals.ts` bridges the OneCLI approval path into
the existing `pending_approvals` flow. See the inherited
`CLAUDE.md` section "Secrets / Credentials / OneCLI" for the
gotcha around new agents starting in `selective` secret mode.

### 6.5 Provider boundary

The agent provider is Claude Code subscription, accessed via
`ANTHROPIC_BASE_URL` pointing at the OneCLI proxy. OneCLI rewrites
the auth header so the container never sees the raw subscription
token. There is no Bedrock fallback (removed in P5). A future
provider swap (e.g. local model via `add-opencode`) is mechanical:
install the provider skill, change `container_configs.provider`
in the central DB.
