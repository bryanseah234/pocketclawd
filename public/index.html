1 Executive Summary

1.1 Project Overview

NanoClaw WhatsApp Assistant is a cloud-native AI-powered WhatsApp assistant system built on Azure infrastructure. The system provides personalized AI assistance to users through WhatsApp messaging, featuring document processing, corporate knowledge retrieval, and automated daily notifications.

1.2 Key Objectives

  • Enterprise-grade AI assistance through WhatsApp with complete data isolation per user
  • Seamless document processing and knowledge base management
  • Automated daily briefings based on user preferences
  • Slide generation capabilities for presentations

1.3 Target Users

The system targets corporate employees in Singapore who require AI-assisted document processing and knowledge retrieval through their personal WhatsApp devices. Users range from knowledge workers who need quick access to corporate information to administrative staff who benefit from automated daily briefings.

2.2 Component Description

2.2.1 NanoClaw Orchestrator

The orchestrator serves as the central nervous system of the entire application. It handles incoming WhatsApp messages through the Baileys bridge, manages the lifecycle of sub-agent containers, maintains the Redis message queue for inter-service communication, and coordinates scheduled tasks such as daily notifications. The orchestrator runs as a systemd service and automatically restarts upon failure.

2.2.2 Sub-Agent Containers

Each user is assigned a dedicated Docker container running a lightweight FastAPI application. Sub-agents are responsible for processing user requests, querying the vector store for RAG, communicating with the LLM, and handling document ingestion. Each container maintains isolated filesystem and network namespaces to ensure data security.

2.2.3 Redis Message Queue

The Redis message queue handles asynchronous communication between the orchestrator and sub-agents. This decoupled architecture allows for graceful handling of high load scenarios and provides resilience against temporary service disruptions.

3.1.3 Disaster Recovery Configuration

For production deployments requiring high availability, the architecture supports deployment across multiple VMs in an availability set. The primary components requiring HA consideration include the orchestrator service which requires stateless design to run on multiple VMs, the Redis message queue which should be replaced with Azure Cache for Redis in HA mode, and the Docker containers which can be spawned on any healthy VM.

3.2 Docker Configuration

3.2.1 Rootless Docker Setup

The system implements rootless Docker as the primary isolation mechanism, combined with a Docker socket proxy to reduce the attack surface of the Docker daemon. This dual-layer approach provides security without sacrificing functionality. Rootless Docker allows the Docker daemon to run as a non-root user, significantly reducing the impact of potential container escape vulnerabilities. The docker-socket-proxy further restricts the Docker API surface by blocking dangerous operations while allowing the necessary functionality for container lifecycle management.

3.2.4 Container Security Hardening

Containers run as non-root user with UID 1000. The root filesystem is read-only except for explicitly mounted tmpfs volumes. All Linux capabilities are dropped, preventing containers from performing privileged operations. Seccomp filtering restricts available system calls to a minimal safe set. New privileges are disabled, preventing any subsequent privilege escalation.

3.3 Storage Architecture

3.3.1 Local Storage

The VM maintains local storage for Docker images, container writable layers, and temporary processing files. Docker data is stored at /var/lib/docker on a dedicated data disk to prevent OS disk exhaustion. Temporary files generated during document processing are stored in tmpfs mounts with a maximum size of one hundred megabytes per container. All temporary files are automatically cleaned up upon container termination.

3.3.2 Azure Blob Storage

All user-uploaded files are streamed directly to Azure Blob Storage, bypassing local VM storage entirely. This approach prevents storage exhaustion on the VM and provides built-in redundancy through Azure's geo-replicated storage. Files are organized under the following container structure: staging (newly uploaded files pending malware scanning), documents (files that have passed security checks and are ready for processing), corporate (corporate documents accessible to all users), and sessions (Baileys authentication sessions for session persistence across VM restarts).

4.1.2 Container Specifications

The chat_messages container stores all conversation history between users and the AI assistant. The partition key is userId, enabling efficient queries for individual user histories. The RU/s allocation is four hundred to accommodate the expected one thousand messages per day across all users. Time-to-live is set to ninety days for conversation retention.

The webhook_tokens container manages one-time tokens for document save confirmations. Each token is hashed with SHA-256 and stored with its hash as the partition key. The TTL is fifteen minutes to automatically expire unused tokens.

The user_preferences container stores all user configuration including notification settings, search preferences, and slide generation preferences. This container has no TTL as preferences are retained indefinitely.

The system_errors container logs all application errors for debugging and monitoring. The TTL is thirty days to balance debugging needs with storage costs.

4.1.3 Throughput Configuration

All containers are configured with provisioned throughput rather than serverless mode. This choice ensures consistent performance and eliminates the cold-start delays associated with serverless databases.

4.2.2 Security Model

Data isolation is enforced through query-time filtering. Every search query automatically includes a filter clause that restricts results to the requesting user's documents or corporate documents. This approach eliminates the need for multiple indexes while maintaining strict data separation. The mandatory filter pattern is userId eq '{user_id}' OR userId eq 'CORPORATE'. This filter is applied programmatically and validated at the application layer to ensure it is never omitted.

4.2.3 Search Configuration

The vector search configuration uses the text-embedding-3-large model from Azure OpenAI with one thousand five hundred thirty-six dimensions. The minimum similarity threshold is set to zero point seven to filter out irrelevant results while maintaining high recall.

5.1.2 Admin QR Code Interface

The admin web interface provides a secure location for QR code scanning during initial setup. When a new session is required, the orchestrator generates a QR code and displays it through the admin web interface. Admins are notified via email when manual intervention is required for authentication. QR codes are only valid for five minutes before expiring. Failed scan attempts are logged for security monitoring. Multiple failed scans trigger an alert for potential unauthorized access attempts.

5.1.3 Session Refresh

WhatsApp sessions typically remain valid for thirty days or longer. The system monitors session validity daily and alerts administrators seven days before expected expiration. If session renewal is required, the admin receives an email notification with instructions to access the admin interface and scan a new QR code.

5.2 Document Save Authorization

5.2.1 Webhook Token Security

Document save confirmations use cryptographically secure one-time tokens that expire after fifteen minutes. Token generation uses the secrets.token_urlsafe function with thirty-two bytes of randomness, providing two hundred fifty-six bits of entropy. The token hash is stored in Cosmos DB using SHA-256 hashing. Tokens are validated using constant-time comparison to prevent timing attacks. Each token is immediately invalidated after single use to prevent replay attacks.

5.2.2 User Data Isolation

Complete isolation between users is achieved through multiple layers of enforcement. At the container level, each user runs in an isolated Docker container with separate filesystem and network namespaces. At the storage level, Cosmos DB queries always include userId in the partition key, ensuring queries never cross user boundaries. At the search level, all Azure AI Search queries include mandatory userId filtering that is validated programmatically before execution.

5.4.2 Malware Scanning

All uploaded files are scanned by Azure Defender for Storage before being processed. The scanning process operates asynchronously with a timeout of five minutes. Files that exceed the timeout are held in quarantine until scanning completes. Files identified as malware are deleted and the uploading user is notified with guidance on safe computing practices.

5.4.3 Content Sanitization

PDF files are checked for encryption and script content. Password-protected PDFs are rejected with instructions to remove the password before uploading. PDFs containing embedded JavaScript or automatic actions are stripped of these elements before processing. Office documents are checked for macros and external links. Documents with macros are processed but users are warned about the security implications of macro-enabled files.

5.5 Prompt Injection Prevention

5.5.1 Input Sanitization

All user inputs and document content are sanitized before being processed by the LLM. The sanitization process removes common prompt injection patterns using regular expression matching against known attack signatures. Control characters and non-printable characters are stripped from all text inputs. Input length is capped at ten thousand characters to prevent buffer overflow attempts.

5.5.2 LLM Guardrails

Azure Content Safety is integrated into the processing pipeline to detect and block harmful content. Both user inputs and LLM outputs are scanned for policy violations including hate speech, violence, and sexually explicit content. Policy violations result in the affected message being blocked and the user receiving an explanatory error message.

6 WhatsApp Integration

6.1 Message Handling

6.1.1 Incoming Message Processing

Incoming WhatsApp messages are processed through the following pipeline. The Baileys library receives the message and forwards it to the orchestrator. The orchestrator validates the message format and extracts the sender's phone number as the userId. Messages are categorized as text, file, image, or command. Text messages are queued for LLM processing. File and image messages are queued for document processing. Commands starting with a forward slash are processed as system commands.

6.1.2 Message Rate Limiting

WhatsApp imposes rate limits on outgoing messages to prevent spam. The system implements a message queue with rate limiting to comply with these constraints. The rate limiter enforces a maximum of twenty messages per minute to any single user and two hundred messages per hour across all users. Messages exceeding these limits are queued and delivered when capacity becomes available. Users are not notified of queuing delays as the delays are typically minimal.

6.1.3 Daily Notification Batching

Daily notifications are sent in batches with a three-second gap between each recipient to avoid triggering WhatsApp rate limits. The batch processor runs during off-peak hours to minimize the impact on regular message processing.

7 Document Processing Pipeline

7.1 Upload Flow

7.1.1 Immediate Session Chat

When a user uploads a document via WhatsApp, the system immediately begins a temporary processing session. The document is streamed directly from WhatsApp to Azure Blob Storage staging area. The user receives an immediate acknowledgment message indicating that processing has started. During this phase, the document content is loaded into the sub-agent's temporary memory context. The user can immediately start chatting about the document while processing continues in the background.

7.1.2 Document Processing Steps

Document processing follows a defined sequence of steps ensuring consistent handling and proper security validation.

  1. File Validation: File downloaded from WhatsApp and validated for format and size. Files exceeding 25 MB are rejected.
  2. MIME Type Validation: Magic byte validation confirms file type matches declared extension. Mismatched files are rejected.
  3. Malware Scanning: Azure Defender scans the file. File remains in staging until scanning completes successfully.
  4. Text Extraction: Document content converted to plain text using type-specific extractors.
  5. OCR (if needed): For image-based PDFs, GPT-4o Vision generates text descriptions of each page.
  6. Chunking: Extracted text split into chunks of 512 tokens with 50-token overlap between chunks.
  7. Embedding: Each chunk converted to vector embedding using text-embedding-3-large with 1536 dimensions.
  8. Indexing: Chunks uploaded to Azure AI Search with userId, filename, document type, and ingestion timestamp metadata.

7.2 Document Ingestion Pipeline

7.2.1 Text Extraction

The text extraction process uses specialized handlers for each file type to maximize extraction quality. For PDF files, PyMuPDF is used for text-based PDFs while GPT-4o Vision handles image-based PDFs. For DOCX files, python-docx extracts text with formatting preservation. For XLSX files, openpyxl extracts cell values and sheet names. For PPTX files, python-pptx extracts text from shapes and notes.

7.2.2 Chunking Strategy

Documents are chunked using a recursive character splitter with the following parameters. The chunk size is 512 tokens, optimized for semantic coherence and LLM context efficiency. The overlap between chunks is 50 tokens (approximately 200 characters at 4 chars/token average), ensuring context continuity at chunk boundaries. Separators are applied in priority order: double newlines, single newlines, spaces, and individual characters.

7.2.3 Embedding and Indexing

Each chunk is converted to a vector embedding using text-embedding-3-large with 1536 dimensions. Embedding generation uses batch processing with a batch size of 50 chunks per API call. Rate limiting is applied with exponential backoff to handle Azure AI Foundry token-per-minute limits. Failed batches are retried up to five times with delays of 1, 2, 4, 8, and 16 seconds.

7.3 Knowledge Base Management

7.3.1 Permanent Storage Options

After initial processing, users are offered the option to permanently save documents to their knowledge base. The confirmation is delivered as a clickable link that expires after 15 minutes. When the user clicks the confirmation link, a webhook request is sent to the orchestrator. The orchestrator validates the token from the URL against the stored hash. If valid, the document is marked as permanent in the vector store. The temporary staging copy is deleted after successful permanent storage.

7.3.2 Auto-Save Configuration

Users can configure automatic permanent storage through their preference settings. When auto-save is enabled, documents are automatically indexed without requiring confirmation clicks. This option is recommended for users who trust the documents they upload.

7.3.3 Document Management Commands

Users can manage their knowledge base through natural language commands. The /list command shows all indexed documents with file names and dates. The /delete [filename] command removes a specific document from the knowledge base. The /update command re-processes and updates an existing document with the latest version.

7.4 Slide Generation

7.4.1 Template System

The slide generation feature produces PowerPoint presentations from document summaries and user requests. The system includes four pre-built templates that users can select from.

  • Corporate: Professional design with blue color scheme, company-appropriate fonts, structured layouts suitable for business presentations.
  • Modern: Emphasizes clean lines and contemporary aesthetics with bold colors and minimal clutter.
  • Elegant: Uses refined typography and subtle animations for high-end presentations.
  • Informative: Prioritizes data visualization with charts, graphs, and structured information hierarchy.

8 AI and RAG Configuration

8.1 Large Language Model Configuration

8.1.1 Model Selection

Modelgpt-4o
Context Window128,000 tokens
Max Output4,096 tokens
Default Temperature0.5 (balanced creativity and precision)
Summary Temperature0.3 (more factual, less creative)
Creative Temperature0.8 (used for slide generation)

8.1.2 Task-Specific Configuration

For conversational chat, the temperature is set to 0.5 with a maximum of 4,096 output tokens. The system prompt emphasizes concise responses under 500 words and clear citation of sources. For document summarization, the temperature is set to 0.3 to prioritize accuracy over creativity. The system prompt instructs the model to extract key points and organize them logically. For slide generation, the temperature is set to 0.8 to enable creative formatting suggestions. The system prompt emphasizes structure and visual hierarchy. For question answering with RAG, the temperature is set to 0.2 to minimize hallucination and maximize factual accuracy. The system prompt instructs the model to answer only from provided context and to clearly indicate when information is not available.

8.1.3 System Prompts

The default system prompt establishes the assistant's identity and behavioral guidelines. You are NanoClaw, an AI assistant helping users through WhatsApp. Keep responses concise and under 500 words unless detailed explanations are requested. Always cite sources when using information from retrieved documents using the format Source: filename.pdf, page X. If the retrieved information does not contain the answer, clearly state that you could not find the information. Never make up facts or cite documents that were not actually retrieved. Use bullet points for lists and numbered lists for sequences. When generating code, include comments and ensure the code is complete and functional.

8.2 RAG Pipeline Configuration

8.2.1 Retrieval Strategy

The retrieval system uses a hybrid approach combining vector similarity search with keyword matching. Vector search uses cosine similarity with a minimum threshold of 0.7. The top candidate chunks are retrieved based on embedding similarity. Keyword search uses Azure AI Search's built-in BM25 ranking for exact term matching. Results from both methods are combined with vector results weighted at 70% and keyword results at 30%. A cross-encoder reranker is applied to the combined results to improve relevance ordering. The top 3 chunks after reranking are provided to the LLM as context.

8.2.2 Context Formatting

Retrieved chunks are formatted for the LLM with clear attribution and source information. The context block includes the chunk text, the original document filename, the page number if available, and the relevance score as an indication of confidence. This formatting allows the LLM to accurately cite sources and understand the provenance of information.

8.2.3 Long-Term Memory

The system maintains conversation history in Cosmos DB, enabling the agent to reference discussions from previous days and weeks. The retrieval process includes the user's conversation history when generating responses. The system retrieves the last 30 messages in chronological order, providing up to 3,000 tokens of conversation context. This history enables the agent to maintain context across multiple sessions and reference previous discussions when relevant.

8.3.1 Embedding Model

Modeltext-embedding-3-large

8.3.2 Chunking Strategy

Documents are chunked using a recursive character splitter with the following parameters. The chunk size is 512 tokens, optimized for semantic coherence and LLM context efficiency. The overlap between chunks is 50 tokens, ensuring context continuity at chunk boundaries. Separators are applied in priority order: double newlines, single newlines, spaces, and individual characters.

8.3.3 Batch Processing

Embedding generation uses batch processing to optimize throughput and reduce API call overhead. The batch size is 50 chunks per API call. Rate limiting is applied with exponential backoff to handle Azure AI Foundry token-per-minute limits. Failed batches are retried up to 5 times with delays of 1, 2, 4, 8, and 16 seconds.

13.1.2 Scaling Triggers

The following conditions trigger capacity review and potential scaling actions. Active user count exceeds 70% of current capacity for 5 consecutive minutes. Average message processing time exceeds 30 seconds. Container count frequently reaches the configured maximum.

13.1.3 Vertical Scaling

Vertical scaling to a larger VM instance provides the simplest capacity increase path. The recommended progression is E16s_v3 to E20s_v3 for additional headroom and to E32s_v3 for maximum single-VM capacity.

13.2 Horizontal Scaling Strategy

Important Architectural Constraint — WhatsApp Session Affinity

The Baileys WebSocket connection is stateful and bound to a single phone number. In a horizontal scaling scenario with multiple VMs behind a load balancer, WhatsApp traffic must be routed to the correct VM that holds the active Baileys session. Without session affinity, routing WhatsApp traffic to a second VM will break the connection.

13.2.1 Sticky Sessions (Load Balancer Configuration)

For horizontal scaling, configure the Azure Load Balancer with sticky sessions based on source IP address. This ensures all traffic from a given WhatsApp user is routed to the same VM for the duration of their session. The load balancer configuration includes: Rule for WhatsApp traffic with source IP affinity, session persistence duration of 8 hours (matches WhatsApp session timeout), and health probe on port 443 with 3-second interval.

13.2.2 Alternative: Dedicated Baileys VM

As an alternative to sticky sessions, designate one VM as the dedicated Baileys host while scaling sub-agent traffic horizontally across other VMs. The WhatsApp handler runs only on the designated Baileys VM, which forwards messages to sub-agents on the scaled VM pool via internal load balancing. This approach eliminates session affinity concerns but introduces a single point of failure for WhatsApp connectivity.

13.2.3 WhatsApp Cloud API Migration (Future)

For stateless horizontal scaling, migrate to WhatsApp Cloud API (Meta's official API). This eliminates the stateful Baileys session entirely as Cloud API is inherently stateless. Estimated migration effort is 2-4 weeks. The architecture is designed to support this migration with minimal code changes.
